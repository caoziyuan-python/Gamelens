# AI 产品 PRD 写作 Prompt · 完整版

> 可复用于任意 AI 产品 · 基于 GameLens v1.2 PRD 三轮迭代经验整理

---

## 一、使用说明

这份 Prompt 的目标是让 AI 输出一份能通过腾讯/字节 AI 产品经理面试官审查的 PRD。经过三轮面试官质疑迭代，覆盖了从「分析工具」到「AI 驱动决策系统」的完整升级路径。

| 使用场景 | 说明 |
|---------|------|
| 写新产品 PRD | 把第三章的产品信息替换为你的产品，其余结构保留 |
| 优化已有 PRD | 把第四章的质疑清单作为 checklist 逐条检查 |
| 面试准备 | 第五章的面试问答框架可直接用于口头回答 |
| Codex/Claude 代码生成 | 配合第六章的 Codex Prompt 使用，PRD 作为上下文输入 |

---

## 二、核心 Prompt（直接复制给 AI 使用）

```
你是一名资深 AI 产品经理，曾在腾讯/字节跳动主导过 AI 产品从 0 到 1 的完整交付。
我需要你帮我撰写一份 PRD，要求能通过以下面试官的质疑：
  1. 你的产品是「AI 驱动决策系统」还是「分析工具加了个 AI 插件」？
  2. 把 LLM 拿掉，产品还能跑吗？
  3. LLM 输出胡说怎么办？你有验证机制吗？
  4. 这个系统用久了会变得更好吗？
  5. 你的产品智能体现在哪里？

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【产品信息（请替换为你的实际内容）】

产品名称：[你的产品名]
产品形态：[Streamlit / Web App / API / 其他]
核心问题：[这个产品解决了什么问题]
目标用户：[谁在用 / 什么场景下用]
数据来源：[用什么数据]
AI 能力：[用了哪些 AI 技术：LLM / NLP / CV / 推荐等]
项目背景：[Capstone / 实习 / 个人项目 / 其他]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【输出要求（严格按照以下结构）】

必须包含以下 10 个章节，每章节要求见下方详细说明：
- 不要泛泛而谈，每个功能必须具体到输入/处理逻辑/输出
- AI 能力部分必须明确定义主从架构（LLM 主引擎 vs Fallback）
- 所有 Insight/建议必须数据驱动，不能是固定模板
- 必须包含 LLM 输出验证机制（至少 3 层）
- 必须包含 Learning Loop 三级演进设计
- 必须包含面试问答准备（5 个核心问题的口述答案）
```

---

## 三、章节结构详细要求

### 第1章：产品概述（最重要）

**必须包含：**

**1.1 产品定位**
- 一句话核心定位（面试必背版）
- 格式：将 [传统做法] 升级为 [AI驱动做法]，通过 [核心技术] 实现 [价值]
- 产品基本信息表（名称 / 形态 / 用户 / 数据 / 技术 / 背景）

**1.2 AI 智能能力三层定义（核心，不能省略）**

必须定义三个层次：
```
层次一：[能力名称]（英文）
  从 [输入] 自动发现/归纳 [输出]

层次二：[能力名称]（英文）
  理解 [复杂输入]，输出 [结构化结果]

层次三：[能力名称]（英文）
  自动生成 [可执行输出]

最终形成：Data → Insight → Decision 的闭环
```

**1.3 LLM 主从架构定义（必须明确，不能含糊）**
```
LLM = Primary Decision Engine（定义它负责什么）
Rule Engine / 传统算法 = Fallback System（定义它为什么存在）
验证层 = Validation Layer（定义它验证什么）
```

**1.4 AI 能力演进路线（V1/V2/V3）**
- V1：当前交付，说明为什么这个设计是有意的（成本/可解释/可交付）
- V2：数据驱动升级，具体说明用什么数据驱动什么改变
- V3：完整 AI 决策系统，说明终态是什么

---

### 第2章：用户场景（3个，每个必须包含 AI 价值）

每个场景**必须**包含：
- **背景**：用户遇到了什么问题
- **操作**：用户怎么使用这个产品
- **AI 价值**：AI 做了什么「人工/传统工具做不到」的事（这一行最重要）
- **决策**：用户基于 AI 输出做了什么决定
- **影响指标**：Retention / Conversion / Rating / 其他（说明为什么）

> ❌ 禁止：场景描述里没有 AI，只有「查看图表」「看结果」

---

### 第3章：核心功能设计（最长，最详细）

**3.1 功能模块总览表**

三列：模块名 | AI角色 | 用户价值

AI角色必须明确标注（Primary / Fallback / Validation / 无AI），不能全写「提供分析」

**3.2 LLM 核心功能（每个功能必须包含）**
```
定位：Primary / Secondary（说明是主功能还是辅助）
输入：具体是什么数据，多少量级
Prompt 设计：写出实际的 Prompt 模板（不能只说「调用 GPT」）
输出：具体格式（JSON / 自然语言 / 结构化表格）
验证：这个功能的输出如何被验证
```

**3.3 LLM 输出验证机制（必须，至少 3 层）**
```
验证层一：与规则/传统算法结果交叉对比
  触发条件 | 系统行为 | 用户提示

验证层二：情感/语义一致性检查
  触发条件 | 系统行为 | 用户提示

验证层三：置信度评分（基于样本量/数据支持度）
  High / Medium / Low 的判定标准

验证层四（可选）：人工抽检 + 记录日志
```

**3.4 Actionable Insights（必须数据驱动，不能是模板）**

每条 Insight 必须包含：
- 来源（LLM 主输出 / 规则 Fallback）
- AI 置信度（High/Medium/Low，来自验证机制）
- 触发条件（什么数据触发了这条建议）
- 数据证据（具体数字）
- 影响指标（绑定业务目标）
- 反馈按钮（Learning Loop）

**3.5 Learning Loop 三级演进**

| 版本 | 实现内容 |
|------|---------|
| V1 收集 | 收集什么，存哪里，字段定义 |
| V2 作用 | 反馈数据具体怎么用，驱动什么变化 |
| V3 优化 | 微调/RAG/权重调整，说明技术路径 |

---

### 第4章：界面设计

必须包含：
- 页面结构表（页面名 | 核心内容）
- LLM 核心功能必须有专属页面（不能藏在「其他」里）
- AI 置信度标注在 UI 中的体现方式
- 产品形态规划（当前 MVP → 未来 SaaS/正式产品）

---

### 第5章：数据流设计

必须包含：
- **Data Schema**：所有核心字段，包含 LLM 输出字段和验证状态字段
  - 必须有：`confidence` / `validation_status` / `source` 字段
- **完整数据流向表**：每个 Step 说明输入/处理/输出
  - LLM 在数据流中的位置必须是 Primary 步骤，不是最后一步的装饰

---

### 第6章：核心指标（Success Metrics）

必须包含：
- 数据质量指标（样本量/覆盖率）
- **AI 准确率指标**（LLM 主题准确率 / 验证通过率）← 不能省
- AI vs 传统方法一致性指标
- Learning Loop 采集指标
- 产品可用性指标

> ❌ 禁止：全是「样本量 ≥ X」，必须有 AI 质量指标

---

### 第7章：MVP 范围

必须包含：
- **V1.0 必须交付**：LLM 核心功能必须在必交付列表（不能是可选）
- **V1.5 可选增强**：时间允许的锦上添花功能
- **V2.0+ 路线图**：AI 升级 + 用户体验升级 + 产品形态升级

> ❌ 禁止：LLM 功能出现在「可选」列表里

---

### 第8章：技术实现

必须包含：
- 技术栈表（层级 | 技术选型，LLM 必须单独一行）
- 代码架构目录树（LLM 相关代码必须有独立目录 `llm/`）
- 关键配置参数（阈值/模型参数，说明数据来源）

> ❌ 禁止：LLM 只出现在 `utils/` 或 `analysis/` 下的一个文件里

---

### 第9章：面试问答准备（必须有，是 PRD 的一部分）

必须包含以下 5 个问题的完整口述答案（100-200字/题）：

| 问题 | 答案必须体现 |
|------|------------|
| Q1：这个项目最难的地方？ | 非结构化数据 → 可执行决策 的难度，说明为什么选 LLM |
| Q2：把 LLM 拿掉还能跑吗？ | 能跑但退化，Fallback 是为了可靠性不是替代 LLM |
| Q3：LLM 胡说怎么办？ | 列出具体验证层次（≥3层），说明置信度机制 |
| Q4：用久了会更好吗？ | Learning Loop 三级演进和数据飞轮逻辑 |
| Q5：智能体现在哪里？ | 对应三层 AI 能力定义，给具体例子 |

---

### 第10章：简历描述（两个方向）

必须包含：
- **AI 产品经理方向**：强调 PRD 设计 / AI 架构决策 / 产品闭环
- **数据科学方向**：强调技术实现 / 模型选型 / 验证机制

每个方向 150-200 字，可直接复制进简历。

> ❌ 禁止：两个方向写一样的内容

## 四、PRD 自检清单（写完后逐条确认）

### AI 产品高度检查

- [ ] **LLM 是主引擎不是插件** — 把 LLM 拿掉，产品还有核心价值吗？有的话 LLM 没有真正核心化
- [ ] **AI 智能三层定义完整** — 发现层 + 抽象层 + 生成层，缺一不可
- [ ] **产品核心定位一句话清晰** — 能用「Data → Insight → Decision」描述你的产品吗？
- [ ] **Actionable Insights 数据驱动** — 建议从数据推理出来，数据变化时建议会变吗？
- [ ] **Learning Loop 有具体路径** — 收集的反馈「具体怎么用」，不能只说「用于优化」

### 工程可信度检查

- [ ] **LLM 验证机制至少 3 层** — 能回答「胡说怎么办」，每层有触发条件和系统行为
- [ ] **Fallback 逻辑明确** — LLM 不可用时系统怎么降级，用户看到什么提示
- [ ] **Data Schema 包含验证字段** — 有 `confidence` / `validation_status` / `source` 字段吗
- [ ] **代码架构有独立 llm/ 目录** — LLM 相关代码独立模块，不是散落的函数
- [ ] **阈值有来源说明** — 数字是基于数据标定的，不是拍脑袋

### 面试场景检查

- [ ] **五道面试题都有答案** — 背出来，不要面试时现想
- [ ] **简历描述两个方向都有** — AI PM 方向和数据科学方向侧重点不同
- [ ] **V2/V3 路线图具体** — 说具体用什么技术实现什么改变，不只是「升级 AI 能力」
- [ ] **Capstone 边界合理** — 并发/实时/分布式是 V2 的事，不要用生产标准要求自己
- [ ] **产品差异化说得清楚** — 能清楚回答「和竞品的区别是什么」

## 五、面试回答通用框架

### 必背核心定位一句话

```
「我这个项目的核心在于，把传统的 [旧做法] 升级为 AI 驱动的决策系统，
 通过 LLM 实现从 [数据输入] 到 [洞察产出] 再到 [产品建议] 的自动化闭环。」
```

**GameLens 示例版：**
> 「把传统评论分析工具升级为 AI 驱动的决策系统，通过 LLM 实现从用户评论到主题洞察再到产品建议的自动化闭环。」

---

### 「最难的地方」框架

```
结构：难点定义 → 为什么难 → 我怎么解决的

「最难的是把 [非结构化输入] 转化为 [可执行输出]，而不只是 [分析结果]。
 [说明为什么难：语义变体/隐含信息/规则无法覆盖]。
 这也是我在 V1 就把 LLM 作为主推理引擎的核心原因，
 而不是传统的 [旧方法]。」
```

---

### 「LLM 胡说怎么办」框架

```
结构：承认问题 → 说出具体解法 → 量化保证

「这是我在设计阶段重点考虑的问题，我设计了 [N] 层验证机制：
 第一层，[验证层名称]：[触发条件] 时 [系统行为]；
 第二层，[验证层名称]：[触发条件] 时 [系统行为]；
 第三层，置信度评分：样本量不足时降低置信度标注。
 所有输出都带 AI 置信度标注（High/Medium/Low），
 让用户知道哪些结论可信，哪些需要人工复核。」
```

---

### 「智能体现在哪里」框架

```
结构：三层 AI 能力 + 每层给具体例子

「体现在三个层次：
 第一，[发现层]——从 [输入] 自动发现 [隐含内容]，
       比如 [具体例子，说明传统方法做不到的地方]；
 第二，[抽象层]——理解 [语义变体]，
       把 [例子A / 例子B / 例子C] 归为同一个问题；
 第三，[生成层]——直接输出 [可执行格式] 的建议，
       用户不需要自己从数据里猜结论。
 核心是：Data → Insight → Decision 的自动化。」
```

---

### 「用久了会更好吗」框架

```
结构：V1 现有 → V2 升级 → 数据飞轮

「V1 已经开始收集数据——每条建议有反馈按钮，记录是否采纳。
 V2 会用低采纳率数据调整 [规则权重/Prompt 设计]。
 V3 用反馈微调 LLM。
 更重要的是，[数据积累] 本身就是壁垒——
 [数据越多] → [洞察越准确] → [建议越好] → [用户越信任]
 形成数据飞轮。」
```

## 六、配套 Codex/Claude 代码生成 Prompt

写完 PRD 后，用以下 Prompt 交给 Codex 生成代码原型。把【产品信息】替换为你的 PRD 内容。

```
你是一名全栈工程师，需要基于以下 PRD 构建产品的代码原型。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【产品信息（从你的 PRD 复制）】

产品名称：[从 PRD 封面复制]
交付形态：[Streamlit / Flask / 其他]
核心功能：[从 PRD 第3章复制]
技术栈：[从 PRD 第8章复制]
Data Schema：[从 PRD 第5章复制]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【代码架构要求】

严格按照 PRD 第8章的目录结构：
- LLM 相关代码必须在独立的 llm/ 目录下
- 每个模块独立文件，每个函数写 docstring
- 配置参数集中在 config.py，不硬编码在业务逻辑里

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【LLM 集成要求】

- LLM 是主推理引擎，不是可选功能
- 每个 LLM 调用必须有 try/except 和 Fallback 逻辑
- 所有 LLM 输出必须经过验证层（见 PRD 第3.3章）
- 验证结果写入 validation_log.csv

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【异常处理要求】

- API 失败：retry 最多 3 次，失败后触发 Fallback
- LLM 验证失败：标注置信度，不阻断流程
- 空数据/边界情况：打印 warning 并 continue，不中断整体流程

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【交付物】

1. 完整可运行的代码（按 PRD 目录结构）
2. requirements.txt
3. README.md（含本地运行说明）
4. 每个关键 Cell/函数上方写 Markdown/docstring 说明分析目的
```

## 七、PRD 迭代经验总结

### v1.0 → v1.1：从「分析工具」到「AI 增强产品」

| 迭代点 | 教训 |
|--------|------|
| 补充产品定位 | 没有用户定位会被问「和竞品有什么区别」 |
| LLM 从可选到核心 | LLM 只做摘要 = 装饰品，会被说「不需要 AI」 |
| Actionable Insights 数据驱动 | 模板建议 = 写死的结论，数据变建议不变是致命问题 |
| 加 Learning Loop | 没有反馈机制 = 系统不会进化，「用久了会更好」答案是「不会」 |
| 每条 Insight 绑定影响指标 | 没有业务目标的建议 = 没有决策价值 |

### v1.1 → v1.2：从「AI 增强产品」到「AI 驱动决策系统」

| 迭代点 | 教训 |
|--------|------|
| 明确 LLM 主从架构 | 「把 LLM 拿掉还能跑」是最毒的问题，必须明确主从 |
| 加 LLM 验证机制 | 「LLM 胡说怎么办」是必问题，没有验证机制是硬伤 |
| AI 智能能力显性化 | 「智能体现在哪里」答不上来 = 不理解自己的产品 |
| 面试问答写进 PRD | PRD 不只给研发看，也是自己的面试答案书 |

### 始终不变的原则

- **Capstone 不需要生产级标准** — 并发/实时/分布式是 V2 的事
- **有意的设计决策比「什么都做」更重要** — V1 用规则引擎是有原因的，说清楚原因
- **每个技术选择都要能回答「为什么」** — 不能只说「用了 XXX」
- **PRD 写给三种人** — 研发（知道做什么）/ 面试官（知道你懂什么）/ 自己（知道怎么回答）
