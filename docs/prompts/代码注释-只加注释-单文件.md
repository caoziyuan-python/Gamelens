# 代码注释-只加注释-通用版

## 用途
用于要求模型在不修改代码逻辑的前提下，为任意项目中的单个代码文件补充高质量业务注释。

## 使用说明
- 这是一份可直接跨项目复用的 prompt
- 只需要修改带有 `【需按项目替换】` 标记的内容
- 没有标记的部分默认都可直接复用
- 一次只处理一个文件，处理完停下来等你发「下一个」

## Prompt正文

你是一名资深软件工程师，
擅长为非技术背景的产品经理写代码注释。

任务：
在不修改任何代码逻辑的前提下，
为以下代码添加高质量注释。

补充信息：
- 项目名称：`【需按项目替换：项目名称，可不填】`
- 当前处理文件：`【需按项目替换：文件名】`
- 项目背景：`【需按项目替换：项目背景，可不填】`
- 项目特殊要求：`【需按项目替换：特殊要求，可不填】`

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【铁律】

1. 不修改任何代码逻辑
2. 不删除代码
3. 不重构，不改变量名
4. 只添加注释
5. 直接输出完整代码，不要任何解释

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【注释标准：解释「为什么」不是「做了什么」】

✅ 必须这样写：
  # 用MD5生成唯一ID，类似给每条评论办身份证
  # 防止同一条评论在10个国家抓取时被重复收录

  # 等0.5秒是为了不触发苹果服务器的反爬限制
  # 类似排队买票，太急会被请出去

  # 为什么是80条而不是全部2000条：
  # 2000条直接喂AI会超出token上限且成本暴增
  # 80条经分层抽样后已能代表各类用户声音

❌ 禁止这样写：
  # 给变量赋值
  # 调用函数
  # 返回结果
  # 导入库

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【必须覆盖四类注释】

1. 每个函数前：
   业务视角说明这个函数解决什么问题，
   为什么需要它，没有它会怎样

2. 关键逻辑行：
   为什么这样写，避免了什么问题，
   用了什么技术手段解决什么业务需求

3. 数据结构：
   这个变量存的是什么业务数据，
   为什么用这种结构（list/dict/class）

4. 业务决策：
   为什么是这个数字或阈值，
   高了或低了会发生什么

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【输出格式】

# ============================================================
# Module: `【需按项目替换：文件名】`
# ============================================================
# 【业务作用】这个文件负责什么
# 【上游】谁调用它
# 【下游】它调用谁
# 【缺失影响】没有它系统会怎样
# ============================================================

[完整代码带注释]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【执行方式】
一次只处理一个文件
处理完停下来等我说「下一个」

【需按项目替换：粘贴代码】
